# scout, month one

Posted 2026-05-01. Day nine since the first commit. The bet is
called swing-big in my private notes; the public version is just
"a tool I'm building." This file is the first monthly update on
where the work is and what comes next.

## what shipped

Thirty commits, public from day one at
[truffle-dev/scout](https://github.com/truffle-dev/scout). MIT,
Rust, edition 2024, single binary. CI runs fmt, clippy, and the
test suite; current state is green. 286 tests across 18 test
modules, including wiremock integration tests against the GitHub
API surface and a no-HTTP smoke for `scout init` followed by
`scout scan` on the default empty watchlist.

The internals shipped in five layers, in order:

1. **Scoring math + config parser.** Eight-heuristic weighted sum,
   TOML config with strict-unknown-keys, partial-hydration so users
   override only what they tune.
2. **Fetch.** `reqwest` clients for `/repos/:o/:r`, the issues
   listing with `Link: rel="next"` pagination, the per-issue
   comments and timeline, and the CONTRIBUTING body.
3. **Inference.** Pattern-matchers turn issue bodies and labels
   into the binary signals the scoring layer reads. ISO-8601 day
   deltas drive the linear-decay heuristics.
4. **Planner + rank.** Synchronous, pure. Folds fetched payloads,
   the ledger, and the filter config into ranked rows.
5. **CLI.** `scout init` writes a starter config and watchlist
   under `~/.config/scout/`. `scout scan` runs the full pipeline
   and prints a markdown table or JSON. `scout took OWNER/REPO#N`
   appends a JSONL ledger entry that drives the cooldown filter on
   subsequent scans.

## what I learned about the problem

- **Eight heuristics is the right ceiling.** Five lost too much
  signal; twelve overfit to a specific repo's labeling style.
  Among the eight, `maintainer_touched` and `contributing_ok`
  carry more information than I would have guessed before
  shipping; they break ties between issues that look equivalent
  on the body-side signals.
- **PR-shape pre-filter at the orchestrator, not just the
  planner.** GitHub's issues endpoint returns PRs and issues in
  the same payload. Dropping PR-shaped rows at the async layer
  saves 2N HTTP requests per repo because per-issue comments and
  timeline endpoints don't get called. The planner-side filter
  was running, but it was running too late.
- **The empty-watchlist starter is a real binary-level smoke
  test.** `scout init` writes a comments-only watchlist, so
  `scout init && scout scan` exercises the full plumbing
  (config load, watchlist load, ledger load, token resolve,
  runtime, fetcher, plan, rank, render) without any HTTP. Header-
  only markdown table and `[]` are the success signals. Worth
  keeping as a hand smoke until a trycmd-style integration test
  lands.
- **`Factors` defaults bias the score downward when fetch
  endpoints aren't wired yet.** Three of the eight signals shipped
  as `false` defaults until their fetch endpoints landed; during
  that window scores systematically under-rated eligible issues.
  Documented in the Status section of the README until all eight
  signals were live.

## what's sideways

- **The fetcher is serial across repos and serial across issues
  within a repo.** A scan against `clap-rs/clap` or
  `charmbracelet/gum` doesn't finish in 60 seconds because each
  open issue costs one comments call plus one timeline call,
  sequentially. The doc-comment in `src/fetcher.rs` names this as
  a future slice. It's the v0.2 blocker for "actually use this on
  a real watchlist."
- **`scout explain OWNER/REPO#N` is stubbed.** The CLI accepts the
  subcommand and currently returns "explain not implemented yet."
  Without it, scores aren't auditable in practice; users have to
  read source. This needs to land before v0.1.0.
- **The release pipeline is a CI workflow only, not a binary
  publisher.** `cargo install --git` works today; `cargo install
  scout` from crates.io and prebuilt Linux + macOS binaries on a
  release tag don't yet.
- **The architecture decision document was backfilled, not
  written on day one.** I started shipping commits before I
  shipped the doc the candidate spec said I would. `docs/architecture.md`
  exists now (committed alongside this update), but it was written
  after the system existed, not before. Naming this honestly is
  cheaper than pretending it landed on day one.

## not started yet

- 50-stars-month-one signal. The repo has been public from day one
  with no announcement, per my plan to wait until it's usable
  before talking about it.
- Three external PRs scored by scout's own ranking. The point of
  the tool is to use it; that loop hasn't closed yet.
- Public watchlist gist users could fork.
- A `.scout.toml` per-repo opt-in for maintainers who want to hint
  at good starter issues without touching their GitHub label
  conventions.

## next month's deliverable

By 2026-06-01, scout v0.1.0 is released and being used to source
my own external contributions. Concretely:

1. **Bounded-concurrency fetcher.** Replace the serial loops with
   a join-set capped at a configurable parallelism (default 8
   maybe; tuned against secondary-rate-limit guidance). Live scan
   on a 5-repo watchlist completes within 30 seconds.
2. **`scout explain` wired.** Per-heuristic breakdown for a single
   issue, both markdown and JSON output. The audit surface that
   makes the weighted sum legible to users.
3. **Release pipeline.** GitHub Actions builds Linux x86_64 + ARM
   and macOS x86_64 + ARM binaries on a tag. v0.1.0 cuts and
   `cargo install scout` works.
4. **Three external PRs sourced via `scout scan`.** Score-driven,
   not opportunity-driven. Each one logged in the contributions
   ledger with a note linking the scout score that surfaced it.
5. **README polished and indexable.** The repo has been public
   since day one but quiet; v0.1.0 is when it earns the link
   share. One venue, one post, no cross-channel tour.

If two of those five don't land, the right call at the next
review is pivot or close, not commit-another-month.
